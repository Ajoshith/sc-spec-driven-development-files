For new interns with coding agents

Stop vibe-coding.
Start with a spec.

Five slides on why writing the plan first makes an AI coding agent faster, not slower — for anything bigger than a one-off script.

Spec-Driven Development · 101
SDD 10101 / 08
The core difference

Two ways to work with an agent

Same tools, same model. What changes is whether the thinking is written down.

✗ Vibe-coding

Improvise turn by turn

  • Decisions live in chat scrollback
  • Next session starts from zero
  • Scope creep is invisible
  • "Why this stack?" → nobody knows
→
✓ Spec-driven

Write the plan, then build

  • Decisions written down, with the "why"
  • Anyone can pick up mid-project
  • Out-of-scope is stated up front
  • Every choice is traceable to a file
SDD 10102 / 08
Step 1 · Foundation

A constitution the whole project answers to

Before any feature: three short files everything else has to obey.

mission.md who & why
tech-stack.md the stack
roadmap.md phase order
→
Every phase checks against these

No re-litigating the stack each feature. Deferred choices (auth, tests, hosting) are named as deferred — not silently guessed.

SDD 10103 / 08
Step 2 · Repeat per feature

The same loop for every unit of work

Find the phase → branch → ask what's ambiguous → write three files → then code.

1

requirements.md

Scope, decisions, what's out

2

plan.md

Small numbered task groups

3

validation.md

Checklist for "done"

4

build & merge

One branch per phase

The agent asks you 3 grouped questions on real forks in the road — instead of guessing wrong and un-guessing three phases later.

SDD 10104 / 08
Step 3 · Compounding

When a process repeats, encode it as a skill

Typed the same instructions 3 times? That's the signal to automate them.

Before

Paste the full "start a phase" instructions every single time. They drift, they get forgotten.

→
skills/next-phase "next phase"
skills/changelog auto after merge

A skill is a spec for a process — say the trigger phrase and the agent runs it the same way, every time.

SDD 10105 / 08
Step 4 · Course correction

Re-plan by editing one file — not renegotiating

Three phases turned out to be one. Because the roadmap is a file, that's a small visible edit.

Phase 3 · Booking
Phase 4 · Roles
Phase 5 · Polish
→
Phase 3 · Booking, roles & polish

Then a fresh spec is derived from the updated roadmap — structure changes with reality, not against it.

SDD 10106 / 08
Step 5 · The record

Every merge leaves a receipt

One CHANGELOG.md entry per phase — dated, plain-language, linked to its spec. Auto-appended by the changelog skill after each merge.

07-09Phase 3 · Booking, roles & polish→ spec
07-09Phase 2 · Read-only dashboard→ spec
07-09Phase 1 · Core data model→ spec
07-09Phase 0 · Project setup→ spec

Answers "what shipped, and when?" in seconds — no git log archaeology. The one artifact a future intern reads first.
